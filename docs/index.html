01 Why a Copilot SDK?

The agent loop, packaged.

Copilot CLI is great in a terminal. The SDK is what you reach for when you want to embed the same agent loop — tools, permissions, MCP, hooks, persistence — into your own app.

  • Same models, same protocol, same tool catalogue as the CLI
  • Stream tokens, intercept tool calls, swap personas mid-conversation
  • Plug in MCP servers (filesystem, GitHub, anything) in one kwarg
  • Resume any conversation by its session_id — across processes, deployments, days
  • Bring-your-own UI: CLIs, web apps, Slack bots, IDE plugins

04 Authentication

Four ways in. Pick the one that matches your deployment.

01

Signed-in user (CLI login)

Run copilot login once. Best for local dev and desktop apps. Uses your personal Copilot subscription.

02

OAuth GitHub App

Each end-user signs in with GitHub; pass github_token on session create. The pattern for multi-tenant web apps and SaaS.

03

Env variables

Set COPILOT_GITHUB_TOKEN, GH_TOKEN, or GITHUB_TOKEN in CI / containers. Zero interactive prompts.

04

BYOK — bring your own key

Plug in OpenAI, Azure AI Foundry, Anthropic, Ollama, vLLM, … via the provider kwarg. No Copilot subscription required.

Enterprise extras: works with EMU accounts and SAML SSO out of the box; OAuth honours org memberships automatically.

06 Custom tools

Hand the agent a Python function — it calls it when needed.

You declare three things and the SDK does the rest:

  1. Schema — a Pydantic model for the arguments. The model name and field docs become the JSON Schema the model sees.
  2. Handler — an async function that receives the parsed args and returns a JSON-serialisable result.
  3. Register — pass tools=[my_tool] when creating the session.
@define_tool pydantic async
tool_signature.py 
class WeatherArgs(BaseModel):
    city: str = Field(description="City name")

@define_tool("get_weather", "Current weather", WeatherArgs)
async def get_weather(args: WeatherArgs) -> dict:
    return await fetch_weather(args.city)

07 Custom agents

Different personas, different tools — same session.

A custom agent is a named persona with its own system prompt and an explicit, scoped tool list. Swap personas mid-conversation to break a task into specialist roles — researcher, code writer, reviewer, security auditor.

RS

Researcher

Tools: grep, glob, view. Read-only by construction.

CW

Code writer

Tools: view, edit, create. Makes the change.

RV

Reviewer

Tools: grep, view. Audits the diff and reports back.

Set agent="name" on session create to pick the starting persona; switch with session.set_agent(…). Example 03 walks through an explicit writer → reviewer handoff.

08 Hooks lifecycle

Intercept the agent loop at every meaningful step.

Hooks are async callbacks the SDK fires around session events. Use them for audit logs, policy enforcement, prompt enrichment, secret redaction, retry logic — anything that has to wrap the agent without changing the model or the tools.

01

on_session_start

Inject context, open an audit trail, attach a request id.

02

on_user_prompt_submitted

Sanitise, redact PII, or enrich the user message before the model sees it.

03

on_pre_tool_use

Allow / deny / modify any tool call. The hard-policy point.

04

on_post_tool_use

Transform tool output — redact secrets, trim noise, log.

05

on_error_occurred

Custom error mapping, retries, fallbacks.

06

on_session_end

Flush metrics, finalise audit logs, close down resources.

09 MCP servers

Plug in any tool that speaks Model Context Protocol.

MCP is an open standard for connecting AI agents to external tools. Point the SDK at any MCP server and its tools appear in the catalogue alongside the built-ins — no per-server glue code.

LO

Local / stdio

The SDK launches a subprocess and talks over stdin/stdout. Great for filesystem, SQLite, Puppeteer, local scripts.

command + args

HT

Remote / HTTP

The SDK connects to an MCP endpoint over HTTP (SSE for streaming). Use for shared services and cloud tools.

type: "http"

Combine with session persistence (a session_id you store on disk) to resume the same conversation later — with the same MCP servers re-attached. Example 05 shows both.

10 The pattern you'll see in every example

Two async context managers. One required handler.

  • CopilotClient spawns the bundled CLI as a subprocess and shuts it down for you.
  • create_session opens a conversation. The session_id can be reused later for resume.
  • on_permission_request is required — even PermissionHandler.approve_all counts.
  • send / send_and_wait kick the agent. The reply arrives via events.
session_pattern.py 
from copilot import CopilotClient
from copilot.session import PermissionHandler

async with CopilotClient() as client:
    async with await client.create_session(
        on_permission_request=PermissionHandler.approve_all,
        model="gpt-4.1",
    ) as session:
        reply = await session.send_and_wait(
            "Hello, Copilot."
        )
        print(reply.data.content)

EX 01 Streaming chat

Token-by-token, event-driven.

The SDK is event-driven: session.send() returns immediately and the model's reply arrives as a stream of typed events.

We match on just the two we care about: AssistantMessageDeltaData for each fragment and SessionIdleData to know when to exit.

streaming=True match / case asyncio.Event
01_simple_chat.py 
async with await client.create_session(
    on_permission_request=PermissionHandler.approve_all,
    model="gpt-4.1", streaming=True,
) as session:
    done = asyncio.Event()

    def on_event(event):
        match event.data:
            case AssistantMessageDeltaData(delta_content=d):
                print(d or "", end="", flush=True)
            case SessionIdleData():
                done.set()

    session.on(on_event)
    await session.send("Explain the SDK in 3 sentences.")
    await done.wait()

EX 02 Custom tools

Hand the agent a Python function.

@define_tool packages a function + a Pydantic schema into a tool the model can call on its own.

The Field(description=…) text is the only hint the model gets — write it like a tiny prompt.

@define_tool pydantic send_and_wait
02_custom_tools.py 
class WeatherParams(BaseModel):
    city: str = Field(description="City, e.g. 'Seattle'")

@define_tool(description="Get the current weather for a city")
async def get_weather(params: WeatherParams) -> dict:
    return {"city": params.city,
            "temp_c": random.randint(-5, 35),
            "condition": "sunny"}

async with await client.create_session(
    on_permission_request=PermissionHandler.approve_all,
    model="gpt-4.1", tools=[get_weather],
) as session:
    reply = await session.send_and_wait(
        "Weather in Tokyo and Berlin?")

EX 03 Custom agents

Multiple personas. One session. Hot swap.

A custom agent is a system prompt + a tool allowlist with a name. Bundle several and pick one with the agent= kwarg.

Switch mid-conversation with session.rpc.agent.select(…) — history is kept, persona is replaced. Verify with rpc.agent.get_current().

custom_agents rpc.agent.select rpc.agent.get_current
03_custom_agents.py 
AGENTS = [
    {"name": "researcher", "tools": ["grep", "view"],
     "prompt": "Explore code. Never modify."},
    {"name": "reviewer", "tools": ["grep", "view"],
     "prompt": "Review for bugs and clarity."},
]

async with await client.create_session(
    custom_agents=AGENTS, agent="researcher", ...) as s:
    await s.send_and_wait("What language is this?")
    await s.rpc.agent.select(AgentSelectRequest(name="reviewer"))
    cur = await s.rpc.agent.get_current()  # proves swap
    await s.send_and_wait("Review 01_simple_chat.py")

EX 04 Hooks

Cross-cutting concerns, without polluting tools.

Hooks fire at well-defined lifecycle points: on_pre_tool_use, on_post_tool_use, on_user_prompt_submitted, …

Return None for "no opinion", or a typed dict to allow, deny, rewrite arguments, or inject context.

on_pre_tool_use on_post_tool_use permissionDecision
04_hooks.py 
async def on_pre_tool_use(input_data, invocation):
    print(f"[pre]  {input_data['toolName']}")
    return None   # or: {"permissionDecision": "deny", ...}

async def on_post_tool_use(input_data, invocation):
    print(f"[post] {input_data['toolName']} done")
    return None

    on_permission_request=PermissionHandler.approve_all,
    model="gpt-4.1",
    hooks={
        "on_pre_tool_use": on_pre_tool_use,
        "on_post_tool_use": on_post_tool_use,

EX 05 MCP + persistence

Plug in a tool server. Resume the chat tomorrow.

MCP is the standard plug-in format for LLM tools — filesystem, GitHub, Slack, dozens more.

Pass a stable session_id on creation and the SDK can resume_session(id, …) later with full history.

mcp_servers session_id resume_session
05_mcp_and_persistence.py 
MCP_SERVERS = {
    "filesystem": {
        "type": "local", "command": "npx",
        "args": ["-y",
                 "@modelcontextprotocol/server-filesystem",
                 WORKDIR],
        "tools": ["*"],
    },
}

# First run — pass our stable session_id
session_ctx = await client.create_session(
    session_id=SESSION_ID, mcp_servers=MCP_SERVERS, ...)

# Later — same conversation, full history
session_ctx = await client.resume_session(
    SESSION_ID, mcp_servers=MCP_SERVERS, ...)

EX 06 Human in the loop

Custom permission + ask_user. You stay in control.

The permission handler is called for every sensitive action — read, write, shell, MCP, network. Approve, reject, or ask a human.

on_user_input_request wires the agent's ask_user tool to your UI — terminal, Slack, anything.

on_permission_request on_user_input_request
06_human_in_the_loop.py 
from copilot.session import PermissionRequestResult

def on_permission_request(request, invocation):
    if request.kind.value == "read":
        return PermissionRequestResult(kind="approve-once")
    answer = input(f"approve {request.kind.value}? [y/N]: ")
    return PermissionRequestResult(
        kind="approve-once" if answer == "y" else "reject")

def on_user_input_request(request, invocation):
    answer = input(f"{request['question']} ")
    return {"answer": answer, "wasFreeform": True}

async with await client.create_session(
    on_permission_request=on_permission_request,
    on_user_input_request=on_user_input_request, ...
) as session: ...

12 Deployment patterns

From your laptop to a horizontally-scaled service.

01

Local CLI (auto-managed)

Default. SDK spawns the CLI as a child process; it dies with your app. Zero config — perfect for scripts and dev.

02

Bundled CLI

Ship copilot-darwin-arm64 / linux-x64 / win-x64.exe alongside your binary. Electron, Tauri, Docker images — no separate install for end-users.

03

Headless server (TCP)

copilot --headless --port 4321. SDKs connect over TCP. Independent lifecycle — the pattern for backend APIs and SaaS products.

04

Horizontally scaled

Multi-client sessions, per-tenant isolation, OpenTelemetry tracing, 40+ streaming events. Production at scale.

22 What changed · 0.1 → 0.2.0

Customizable prompts, distributed tracing, in-memory attachments.

01

Fine-grained system prompt

New customize mode edits 10 named sections of the Copilot prompt — identity, tone, safety, … with replace, append, prepend, remove, or a transform callback.

02

OpenTelemetry, every SDK

Set telemetry={"otlp_endpoint": …} and the CLI emits W3C trace context across create, resume, send, and tool handlers.

03

Blob attachments

Send images / binary content inline — no temp files, no disk writes. Useful for screenshots and generated images already sitting in memory.

04

Pre-select a custom agent

Choose the active agent at create_session instead of calling agent.select as a separate round trip.

05

New RPC + bug fixes

Direct session.rpc.* surfaces for skills, MCP, and permission state; broad cross-language consistency pass.

!!

Breaking changes

Naming cleanup in Python and Go SDKs — pin github-copilot-sdk>=0.2.0 and read the release notes before upgrading existing code.

23 What changed · 0.2 → 0.3.0

Multi-tenant auth, agent-scoped tools, scoped permissions.

01

Per-session GitHub auth

Each session can carry its own github_token. Different users, plans, and quotas can coexist on one CLI process — the multi-tenant primitive for SaaS apps.

02

Per-agent excluded_tools

Hide tools from the default agent while keeping them available to sub-agents — clean orchestrator pattern where the parent delegates instead of doing.

03

Per-agent skills (opt-in)

Custom agents declare skills=[…] to inject specific skills at startup. Sub-agents do not inherit parent skills — no surprise context bloat.

04

Sub-agent streaming

Streaming deltas now fire for sub-agents too, each tagged with an agent_id. Configurable session idle timeout replaces the old 30-minute hard limit.

05

Scoped permissions

Return approve-for-session or approve-for-location to persist decisions. Outcomes renamed: approved → approve-once, denied → reject.

06

API polish for GA

Dedicated event types per kind, MCP CallToolResult helper, custom BYOK request headers, casing fixed (githubToken). The runway into 1.0.
